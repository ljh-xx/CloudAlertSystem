# CloudAlertSystem 前端功能优化技术报告

> 版本：v1.0

## 一、项目背景与目标

### 1.1 背景

CloudAlertSystem 是一套期货云端预警系统，当前已实现基础的价格预警和时间预警功能。随着功能迭代，前端交互体验和功能完整性需要进一步优化。

### 1.2 优化目标

1. **用户体验优化**：分离登录流程、优化界面布局、提升操作直观性
2. **功能增强**：价格预警增加区间类型、整合新增/编辑交互
3. **信息展示增强**：列表显示最新价格、合约下拉选择
4. **代码可维护性**：优化前端架构，分离职责

## 二、预警功能定义

### 2.1 预警类型总览

系统支持两类并列的预警规则，二者相互独立：

| 类型   | 触发条件          | 核心参数                  | 典型用途                    |
| ---- | ------------- | --------------------- | ----------------------- |
| 价格预警 | 合约价格满足设定条件时触发 | 合约代码、触发条件、目标价格（或价格区间） | 价格突破提醒、区间入场提醒           |
| 时间预警 | 系统时间到达设定时刻时触发 | 合约代码、触发时间（HH:MM:SS）   | 开盘/收盘提醒、数据发布提醒、交易计划定点提醒 |

**共性特征**：

- 触发后状态均变为「已触发」，仅触发一次
- 通知渠道一致：客户端在线则推送消息，离线则发送邮件
- 持久化机制一致：均存储于数据库，支持状态查询与重启恢复
- 状态一致：未触发（Pending）、已触发（Triggered）、已删除（Deleted）

### 2.2 价格预警条件

价格预警支持三种触发条件：

| 条件码 | 条件名称 | 说明                 | 所需参数        |
| --- | ---- | ------------------ | ----------- |
| 0   | 大于等于 | 最新价 ≥ 目标价时触发（向上突破） | 目标价         |
| 1   | 小于等于 | 最新价 ≤ 目标价时触发（向下突破） | 目标价         |
| 2   | 位于区间 | 最新价进入目标价格区间时触发     | 区间下限价、区间上限价 |

**触发规则**：

- 均为「首次穿越触发一次」，触发后预警状态变为「已触发」，不再重复判断
- 区间条件采用闭区间 `[下限, 上限]`，即 `价格 ≥ 下限 且 价格 ≤ 上限`
- 触发后通知逻辑与现有保持一致

### 2.3 时间预警定义

**核心定义**：时间预警不以期货合约的价格涨跌为触发条件，而是以用户预设的具体时间点作为触发依据，当系统时间到达设定时刻时，自动针对指定期货合约向用户推送提醒通知。

**本质**：定点交易提醒，解决用户「在特定时间关注特定合约」的需求。

**核心要素**：

- **客户账号**：标识预警单的所属用户，用于触发后定向推送通知
- **合约代码**：指定该预警关联的期货合约，通知时明确告知对应合约
- **触发时间**：预警的唯一触发条件，精确到时分秒（HH:MM:SS）

**触发与通知逻辑**：

- 服务器端每秒校验系统时间，到达设定时刻时判定为「已触发」
- 不依赖 CTP 行情接口，仅靠系统时钟完成判断
- 通知规则与价格预警完全一致

**典型应用场景**：

1. **交易时段节点提醒**：如 9:00 日盘开盘、14:55 收盘前、21:00 夜盘开盘
2. **事件/数据时点提醒**：重要宏观数据公布、行业持仓报告发布时间点
3. **交易计划定点提醒**：用户计划在某个时间点观察走势、决定是否下单

## 三、前端交互设计

### 3.1 登录流程优化

#### 现状问题

- 登录控件与主功能控件在同一窗口，登录前所有功能可见但禁用
- 缺乏「先登录、后使用」的仪式感
- 顶部区域拥挤，登录、注册、设置邮箱混在一起

#### 优化方案

**独立登录对话框**：

- 应用启动时先弹出 `CLoginDlg` 登录对话框
- 对话框包含：用户名、密码输入框，「登录」「注册」按钮
- 登录成功后关闭对话框，显示主窗口
- 登录失败停留在登录框，密码清空

**登录后顶部用户栏**：

- 左侧：应用标题 + 当前用户名
- 右侧：连接状态指示 + 「登出」按钮
- 登出后回到登录对话框

### 3.2 主界面布局

#### 整体布局

### 3.3 新增/编辑预警面板

#### 模式说明

右侧面板为**同一个面板，两种工作模式**：

## 四、技术实现方案

### 4.1 协议层改动

#### 新增条件码

```cpp
// Protocol.h
#define COND_GE             0   // last_price >= trigger_price
#define COND_LE             1   // last_price <= trigger_price
#define COND_BETWEEN        2   // last_price between [low, high]  (新增)
```

#### ADD_PRICE 命令扩展

**原格式**：

```
ADD_PRICE <username> <contract> <price> <condition>
```

**新格式**（兼容旧版）：

```
// 条件为 0 或 1 时（大于等于/小于等于）
ADD_PRICE <username> <contract> <price> <condition>

// 条件为 2 时（位于区间）
ADD_PRICE <username> <contract> <price_low> <price_high> <condition>
```

*或者统一为 5 参数格式，条件为 0/1 时忽略 price_high，具体实现时选择更简洁的方案*

#### MODIFY_PRICE 命令扩展

与 ADD_PRICE 保持一致，增加对区间条件的支持。

#### QUERY 返回格式扩展

**原格式**：

```
ALERT <id> <type> <contract> <price> <cond> <time> <status> <created>
```

**新格式**：

```
ALERT <id> <type> <contract> <price> <cond> <time> <status> <created>
```

*price 字段：条件为区间时存下限，上限需要新增字段或用其他方式传递*

**建议方案**：扩展 ALERT 行格式，增加 price_high 字段：

```
ALERT <id> <type> <contract> <price> <price_high> <cond> <time> <status> <created>
```

#### 新增价格查询命令（可选，用于最新价显示）

```
// 请求
GET_PRICE <contract>

// 响应
PRICE <contract> <last_price>
```

或批量查询：

```
// 请求
GET_PRICES <contract1> <contract2> ...

// 响应
PRICE <contract> <last_price>
...
END
```

### 4.2 数据库改动

#### 表结构变更

**alerts 表新增字段**：

```sql
ALTER TABLE alerts ADD COLUMN trigger_price_high REAL DEFAULT 0;
```

**字段说明**：

- `trigger_price`：价格预警的目标价（条件为区间时存下限）
- `trigger_price_high`：价格预警的区间上限（条件为 0/1 时为 0 或 NULL）

#### 数据迁移

- 现有数据的 `trigger_price_high` 默认为 0
- 条件为 0 或 1 的预警，`trigger_price_high` 字段无意义

#### 接口改动

**AlertDB 类需要修改的接口**：

```cpp
// 新增价格预警（增加 priceHigh 参数）
int AddPriceAlert(int userId, const std::string& contract,
                  double price, double priceHigh, int condition);

// 修改价格预警（增加 priceHigh 参数）
bool ModifyPriceAlert(int alertId, int userId,
                      const std::string& contract,
                      double price, double priceHigh, int cond);

// 查询结果结构 AlertRecord 增加字段
double trigger_price_high;  // 新增
```

### 4.3 服务端改动

#### ClientSession

- `HandleAddPrice`：解析新增的 price_high 参数
- `HandleModifyPrice`：解析新增的 price_high 参数
- `HandleQuery`：返回结果中增加 price_high 字段
- 可选：`HandleGetPrice`：新增价格查询接口

#### AlertEngine

- `CheckPriceAlert`：增加区间条件的判断逻辑
  
  ```cpp
  else if (rec.condition == COND_BETWEEN && 
         lastPrice >= rec.trigger_price && 
         lastPrice <= rec.trigger_price_high) {
    fired = true;
  }
  ```

#### CTPMdHandler（可选）

- 维护最新价格缓存，供价格查询接口使用

### 4.4 客户端改动

#### 架构调整

- 新增 `CLoginDlg` 登录对话框类
- `CMainFrame` 中分离登录逻辑和主界面逻辑

#### UI 控件变更

- 移除顶部登录相关控件（移到登录对话框）
- 新增用户信息栏（用户名、连接状态、登出按钮）
- 列表工具栏：状态筛选移到列表顶部
- 右侧面板重构：
  - 新增预警类型单选按钮
  - 合约输入改为可编辑下拉框
  - 价格输入框改为动态显示（目标价 / 下限价+上限价）
  - 按钮改为动态显示（添加 / 保存修改+取消）
- 列表新增「最新价」列

#### 新增功能

- 30秒定时器：刷新最新价格
- 面板模式切换逻辑
- 动态显示/隐藏控件逻辑

3. **价格查询性能**：如果列表中合约较多，批量查询需要优化
4. **MFC 动态控件**：动态显示/隐藏控件需要注意布局重绘问题
